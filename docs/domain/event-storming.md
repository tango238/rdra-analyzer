# Event Storming

> Phase 2 / DDD — rdra-analyzer 自身のドメイン。Core は「RDRA 要求モデル抽出」（[discovery.md](discovery.md) 参照）。

## Actor

- **PdM**（人間）— 解析の起点・業務フローの確定者
- **System**（rdra-analyzer）— 抽出・照合・確定/棄却・合成の自動実行者
- **loop-e2e**（外部システム）— 実績シナリオの供給元／承認済み業務フローの引き渡し先

## Domain Events（時系列・過去形）

### A. 取り込み・解析（インプット系）
1. `解析が起動された` — PdM が対象リポジトリ群を指定
2. `コンテキストが構築された` — CLAUDE.md/AGENTS.md/マニフェスト読込
3. `ルート（APIエンドポイント）が抽出された`
4. `モデル（エンティティ）が抽出された`
5. `画面が分析された` — UI要素・遷移
6. `チェックポイントが保存された` — 中間保存（再開可能に）

### B. 要求モデル抽出（★Core）
証拠の連鎖：**APIエンドポイント → Controller 特定 → 使用モデル特定 ＋ 画面突き合わせ → UC候補**。
鎖が切れる（Controller/モデル不明）UC は棄却される。

7. `UC候補が生成された` — LLM 駆動
8. `コード証拠と照合された` 🎯 ← 正確性の防御線
9. `UCが確定された` 🎯 ← 根拠あり（確定者＝System／自動）
10. `UCが棄却された` 🎯 ← 根拠なし → **棄却ログへ記録**（破棄しない）
11. `関連情報が再導出された`（enrich）🎯 ← 決定的紐付け。画面↔エンドポイント照合を含む

### C. 外部実績の突き合わせ（reconcile）
12. `実績シナリオが取り込まれた` — loop-e2e
13. `棄却UCが実績で救済された` 🎯 ← 「実績由来」確定
14. `新規UCが実績から生成された` — `UC-LE-NNN`
15. `コードと実績の矛盾が検出された` 🎯 ← **コードを真**・要調査フラグ

### D. モデル合成・可視化（アウトプット系）
16. `システム境界が特定された` — 接点（画面）×起点（APIエンドポイント）の対応から
17. `情報モデルが生成された`
18. `状態遷移が生成された`
19. `ビジネスポリシーが生成された`
20. `CRUDギャップが分析された`
21. `ビューワーが生成された`

### E. 業務フロー協働（PdM 承認ループ）
業務フローは「コードから想定」までは自動だが、**最後は PdM が承認して確定**する。

22. `業務フローが想定された` — 確定 UC 群から業務の流れを推定（自動）
23. `業務フローがレビューされた` — PdM
24. `業務フローにフィードバックされた` — PdM（→ 再想定へ差し戻し）
25. `業務フローが再想定された` — System（自動で作り直す）
26. `業務フローが編集された` — PdM（手で微調整）
27. `業務フローが承認された` ✅ — PdM（＝業務フロー確定）
28. `承認済み業務フローが引き渡された` — → loop-e2e（テスト作成・実行は委譲）

## Aggregate 候補

解析セッション / APIエンドポイント / エンティティ / 画面 / **ユースケース** / 棄却ログ / 実績シナリオ / **業務フロー** / システム境界 / CRUDギャップ / ビジネスポリシー / 状態遷移 / **矛盾（要調査 / `Conflict`）← sync #4 で実装、Phase 5 で Aggregate 化を判断**

## 発見された問題点・重要な気づき（赤付箋）

- **「確定」には2種類ある**：UC/エンティティ＝コード証拠による自動確定／業務フロー＝PdM 承認による人間確定。確定者の違いがコンテキスト境界を示唆する。
- **システム境界は新規生成ではなく既存照合の意味づけ**：`enrich` の画面↔エンドポイント突き合わせが、副産物としてシステム境界（アクターとシステムの接点）を描いている。
- **テスト生成・実行はスコープ外**：このツールの責務は「業務フロー生成＋PdM 承認」まで。テスト作成・E2E 実行は loop-e2e。境界を越えないこと。
- **棄却ログは破棄禁止**：Precision 重視で棄却するが、検証フロー（実績救済を含む）の入力として保持する。→ ✅ sync #1 で `rejection_log.json`（理由＋欠落証拠）として保持実装（`analyze --strict`）。✅ 実績救済による再昇格も配線済み（reconcile が名前一致で再昇格・sync #1 follow-on）。

### 残差・違反・命名ズレ

- **🔴 13 救済フロー未配線**: 棄却ログを reconcile が入力に再昇格する経路が無い（openQuestion 既出）。
- **🔴 22–28 業務フロー協働**: 丸ごと未実装。**sync #5（deferred, Phase 5/9–11 が前提）**。
- **🟡 8 照合**: 「APIエンドポイント→Controller→モデル＋画面」の証拠連鎖を検証する単一ステップが無く、抽出＋enrich＋#1 に分散。
- **既定挙動は Recall 寄り**: 棄却(#1)・矛盾(#4)は opt-in / 別レポートで、デフォルトでは確定モデルを変えない（モデルは Precision を to-be とする）。
- **構造ズレ**: コードの継ぎ目（Source/UseCase/Scenario/Reconciliation）は A–E イベント群とズレ。**sync #7（deferred, Phase 4 mapping 前提）**。新規 `Conflict` は Aggregate 候補へ追記すべき。

> 残差はすべて sync 台帳（[sync.md](sync.md)）に追跡済み。新規の取り込み先は `Conflict` Aggregate の明示化（→ Phase 5）。
