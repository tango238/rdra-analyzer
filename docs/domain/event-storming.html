Event Storming

Phase 2 / DDD — rdra-analyzer 自身のドメイン。Core は「RDRA 要求モデル抽出」（discovery.md 参照）。

Actor

  • PdM（人間）— 解析の起点・業務フローの確定者
  • System（rdra-analyzer）— 抽出・照合・確定/棄却・合成の自動実行者
  • loop-e2e（外部システム）— 実績シナリオの供給元／承認済み業務フローの引き渡し先

Domain Events（時系列・過去形）

A. 取り込み・解析（インプット系）

  1. 解析が起動された — PdM が対象リポジトリ群を指定
  2. コンテキストが構築された — CLAUDE.md/AGENTS.md/マニフェスト読込
  3. ルート（APIエンドポイント）が抽出された
  4. モデル（エンティティ）が抽出された
  5. 画面が分析された — UI要素・遷移
  6. チェックポイントが保存された — 中間保存（再開可能に）

B. 要求モデル抽出（★Core）

証拠の連鎖：APIエンドポイント → Controller 特定 → 使用モデル特定 ＋ 画面突き合わせ → UC候補。 鎖が切れる（Controller/モデル不明）UC は棄却される。

  1. UC候補が生成された — LLM 駆動
  2. コード証拠と照合された 🎯 ← 正確性の防御線
  3. UCが確定された 🎯 ← 根拠あり（確定者＝System／自動）
  4. UCが棄却された 🎯 ← 根拠なし → 棄却ログへ記録（破棄しない）
  5. 関連情報が再導出された（enrich）🎯 ← 決定的紐付け。画面↔エンドポイント照合を含む

C. 外部実績の突き合わせ（reconcile）

  1. 実績シナリオが取り込まれた — loop-e2e
  2. 棄却UCが実績で救済された 🎯 ← 「実績由来」確定
  3. 新規UCが実績から生成された — UC-LE-NNN
  4. コードと実績の矛盾が検出された 🎯 ← コードを真・要調査フラグ

D. モデル合成・可視化（アウトプット系）

  1. システム境界が特定された — 接点（画面）×起点（APIエンドポイント）の対応から
  2. 情報モデルが生成された
  3. 状態遷移が生成された
  4. ビジネスポリシーが生成された
  5. CRUDギャップが分析された
  6. ビューワーが生成された

E. 業務フロー協働（PdM 承認ループ）

業務フローは「コードから想定」までは自動だが、最後は PdM が承認して確定する。

  1. 業務フローが想定された — 確定 UC 群から業務の流れを推定（自動）
  2. 業務フローがレビューされた — PdM
  3. 業務フローにフィードバックされた — PdM（→ 再想定へ差し戻し）
  4. 業務フローが再想定された — System（自動で作り直す）
  5. 業務フローが編集された — PdM（手で微調整）
  6. 業務フローが承認された ✅ — PdM（＝業務フロー確定）
  7. 承認済み業務フローが引き渡された — → loop-e2e（テスト作成・実行は委譲）

発見された問題点・重要な気づき（赤付箋）

  • 「確定」には2種類ある：UC/エンティティ＝コード証拠による自動確定／業務フロー＝PdM 承認による人間確定。確定者の違いがコンテキスト境界を示唆する。
  • システム境界は新規生成ではなく既存照合の意味づけ：enrich の画面↔エンドポイント突き合わせが、副産物としてシステム境界（アクターとシステムの接点）を描いている。
  • テスト生成・実行はスコープ外：このツールの責務は「業務フロー生成＋PdM 承認」まで。テスト作成・E2E 実行は loop-e2e。境界を越えないこと。
  • 棄却ログは破棄禁止：Precision 重視で棄却するが、検証フロー（実績救済を含む）の入力として保持する。→ ✅ sync #1 で rejection_log.json（理由＋欠落証拠）として保持実装（analyze --strict）。🟡 実績救済による再昇格の配線は未実装（残課題）。
